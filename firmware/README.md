# Urban Micro Farming - IoT Firmware

Embedded C firmware for the Urban Micro Farming IoT devices. Communicates with a Mosquitto MQTT broker to integrate with the Spring Boot backend.

## Tech Stack

- **C Language** (embedded firmware)
- **PlatformIO** (IDE + build system)
- **MQTT Protocol** (communication with broker)
- **Mosquitto Broker** (MQTT message broker)
- **Arduino Mega2560** (primary microcontroller)
- **ESP8266** (WiFi module)
- **Native Testing** (C unit tests)

## Hardware Setup

- Arduino Mega2560 board
- Arduino Uno Multifunctional Expansion Shield
- ESP8266 WiFi module (WiFi + MQTT connectivity)
- KY-018 LDR module (light sensor)
- DHT11 temperature & humidity sensor
- Soil moisture sensor
- SG90 Servo (motor control)

See detailed hardware documentation in the `doc/` folder.


### 2. Build Firmware
```bash
# Compile for Arduino Mega2560
pio run

# Compile for specific environment
pio run -e mega2560
```

### 3. Run Unit Tests
```bash
# Run native C tests
pio test -e native
```

### 4. Upload to Hardware
```bash
# Build and upload to connected device
pio run -t upload -e mega2560
```

```bash
# Open serial monitor (115200 baud)
pio device monitor --baud 115200
```

## Broker

The Mosquitto MQTT broker runs in a Docker container on an Azure VM.

**Broker address:** `20.240.208.122:1883`

### Start broker locally (development)
```bash
cd iot/broker
sudo docker-compose up -d
```

### Check status
```bash
sudo docker ps
sudo docker logs mosquitto
```

### Stop broker
```bash
sudo docker-compose down
```

## Device Registration

Before publishing any MQTT messages, the IoT device must register with the backend once at boot via HTTP POST to obtain a `setup_id`. All subsequent communication uses MQTT only.

**Request:**
```
POST /api/devices/register
Content-Type: application/json

{
  "serial_number": "UMF-001"
}
```

**Response:**
```json
{
  "setup_id": 1
}
```

The device stores `setup_id` in memory and uses it to construct all MQTT topics. On power loss, it re-registers at next boot.

---

## MQTT Topics

All topics use the pattern `farm/{setup_id}/...`, where `setup_id` is assigned by the backend at registration.

| Topic | Direction | Purpose |
|---|---|---|
| `farm/{setup_id}/inf` | IoT → Backend | Periodic sensor readings (all sensors bundled) |
| `farm/{setup_id}/cmd` | Backend → IoT | Actuator commands |
| `farm/{setup_id}/status` | IoT → Backend | Heartbeat and connection status (LWT) |

---

## Message Schemas

### Sensor reading (`farm/{setup_id}/inf`)

Published every 60 seconds. All values are integers. Temperature and humidity are multiplied by 10 to preserve one decimal place (the AVR cannot handle floats). The backend divides by 10 to get the actual value. Timestamp is generated by the backend, not the device.

```json
{
  "setup_id": 1,
  "sensor_id": null,
  "temperature": 225,
  "humidity": 650,
  "light": 800,
  "soil_moisture": 400
}
```

| Field | Type | Unit | Notes |
|---|---|---|---|
| setup_id | integer | — | Assigned at registration |
| sensor_id | integer or null | — | null until multiple sensors per type are used |
| temperature | integer | °C × 10 | 225 = 22.5°C |
| humidity | integer | % × 10 | 650 = 65.0% |
| light | integer | 0–1023 (ADC) | Raw value |
| soil_moisture | integer | 0–1023 (ADC) | Raw value |

---

### Actuator command (`farm/{setup_id}/cmd`)

Published by the backend to control actuators.

```json
{
  "actuator": "water_pump",
  "action": "on",
  "duration_ms": 5000
}
```

| Field | Allowed values |
|---|---|
| actuator | `water_pump`, `led`, `servo` |
| action | `on`, `off` |
| duration_ms | 0–60000 |

---

### Status / Heartbeat (`farm/{setup_id}/status`)

Published every 30 seconds by the IoT device. Also sent automatically by the broker as Last Will and Testament (LWT) if the device disconnects unexpectedly.

```json
{
  "setup_id": 1,
  "status": "online"
}
```

Possible values for `status`: `online`, `offline`.

## MQTT Connection Configuration

```
MQTT_BROKER_HOST=20.240.208.122
MQTT_BROKER_PORT=1883
MQTT_CLIENT_ID=esp8266-farm-001
MQTT_KEEPALIVE=60
MQTT_QoS=1
```

**Important:** The backend must connect with `clean_session = false` so the broker retains QoS 1 messages during backend downtime and delivers them upon reconnect.

## Testing the Broker

Subscribe to all farm topics:
```bash
mosquitto_sub -h 20.240.208.122 -t "farm/#" -v
```

Publish a test sensor reading:
```bash
mosquitto_pub -h 20.240.208.122 -t "farm/1/inf" -m '{"setup_id":1,"temperature":225,"humidity":650,"light":800,"soil_moisture":400}'
```

## Notes

- Port 1883 is used without TLS. AES-128 application-layer encryption is used as a compensating control.
- `allow_anonymous true` is set for development. In production this must be set to `false` with a password file.
- Any changes to topic structure or message schemas must be approved by all team leads before implementation.
